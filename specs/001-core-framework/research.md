# Research: 001-core-framework

## Config Strategy: YAML → C++ Build-Time Codegen

**Decision**: YAML is the developer-facing authoring format. A PlatformIO `extra_scripts`
Python script converts `devices/<device>.yml` to `src/config/generated_config.h` at build
time. The device never parses YAML at runtime.

**Rationale**: RP2040 has 264 KB SRAM. Runtime YAML parsing (libyaml via YAMLDuino) is
feasible but introduces non-trivial memory risk for a framework that must leave headroom
for user modules. ESPHome uses the same build-time codegen pattern. Zero runtime overhead;
config is baked into flash as `const` structs.

**Runtime override path**: Per-device values that vary without reflashing (broker
credentials, cert filenames) are stored as JSON on LittleFS and merged at boot.
ArduinoJson uses a `StaticJsonDocument<N>` with a fixed stack-allocated pool — precise
memory control, no heap fragmentation.

**Alternatives considered**:
- YAMLDuino runtime parsing — viable, but heavy; rejected on SRAM constraint
- JSON-only (no YAML) — rejected; YAML is the stated developer UX requirement

**Libraries**:
- `bblanchon/ArduinoJson` — runtime JSON overrides
- Build-time codegen: pure Python (PyYAML) in `scripts/gen_config.py`, no device lib needed

## TLS over W5500 Ethernet

**Decision**: `mobizt/ESP_SSLClient` as the TLS layer wrapping `EthernetClient` (W5500).

**Rationale**: ESP_SSLClient explicitly supports RP2040, is actively maintained (2024),
supports configurable BearSSL RX/TX buffer sizes (important for 264 KB SRAM), and has
confirmed W5500 compatibility. The OPEnSLab SSLClient is more widely documented but has
lower activity and does not officially list RP2040 in its support matrix.

**Certificate provisioning**: CA trust anchor stored in flash as a `const` C array
generated by `scripts/gen_certs.py` (using the cert → BearSSL TA tool bundled with
ESP_SSLClient). Client certificates for mTLS (if needed) stored on LittleFS and loaded
at boot — allows per-device cert provisioning without reflashing.

**Alternatives considered**:
- OPEnSLab/SSLClient — widely used but RP2040 support uncertain; rejected
- mbedTLS direct — too heavy and low-level for an Arduino framework; rejected

## MQTT Client

**Decision**: `knolleary/PubSubClient` paired with ESP_SSLClient.

**Rationale**: PubSubClient is the de-facto standard for embedded MQTT. It accepts any
`Client` implementation, so it composes directly with ESP_SSLClient. Simple, minimal,
reliable.

**Critical configuration**:
- MUST call `mqttClient.setBufferSize(1024)` at runtime (not `#define MQTT_MAX_PACKET_SIZE`)
  — the `#define` approach does not reliably override the compiled constant when using the
  library as a dependency
- `setKeepAlive(60)` and `setSocketTimeout(15)` tuned for embedded reconnect behavior
- HA Discovery payloads exceed the 256-byte default; `setBufferSize` is non-negotiable

**Alternatives considered**:
- arduino-mqtt (256dpi) — cleaner API, MQTT 5 support, but less community adoption; may
  consider for a future MQTT 5 upgrade path
- MQTTPubSubClient_Generic — explicitly RP2040 + W5500 tested; viable fallback if
  PubSubClient proves problematic

## Host-Side Unit Testing

**Decision**: PlatformIO `[env:native]` + Unity test framework + `ArduinoFake` for Arduino
core mocking. HAL abstraction via C++ pure virtual base classes with constructor injection.

**Rationale**: `ArduinoFake` (FabioBatSilva) mocks the entire Arduino core (`digitalWrite`,
`delay`, `Wire`, `Serial`, etc.) using FakeIt. Combined with a custom `IGPIODriver`
interface, business logic is fully testable on host without hardware. This is the
established pattern in the PlatformIO community.

**Architecture discipline**: All business logic (reconnect state machine, config parsing,
module orchestration) lives in pure C++ classes receiving HAL via constructor injection.
`setup()`/`loop()` in `main.cpp` is wiring only — not tested on native.

**Libraries**:
- `ArduinoFake` — Arduino core mocking
- Unity — built into PlatformIO, no extra lib needed

## RP2040 Platform & Board

**Decision**: `maxgerhardt/platform-raspberrypi` wrapper with `earlephilhower/arduino-pico`
core. Board ID: `silicognition_rp2040_shim`.

**Rationale**: The earlephilhower core is the most complete and actively maintained Arduino
platform for RP2040 — full LittleFS, PIO, USB, SPI/I2C support. The maxgerhardt wrapper
integrates it cleanly into PlatformIO's build system.

**Flash partition**: `board_build.filesystem_size = 1m` — 1 MB LittleFS for runtime
config/certs, 1 MB firmware. Adjust if firmware grows beyond 1 MB.

**`platformio.ini` baseline**:
```ini
[env:rp2040_shim]
platform = https://github.com/maxgerhardt/platform-raspberrypi.git
framework = arduino
board = silicognition_rp2040_shim
board_build.core = earlephilhower
board_build.filesystem_size = 1m
lib_deps =
    knolleary/PubSubClient
    mobizt/ESP_SSLClient
    bblanchon/ArduinoJson
monitor_speed = 115200

[env:native]
platform = native
lib_deps =
    ArduinoFake
test_build_src = yes
```
